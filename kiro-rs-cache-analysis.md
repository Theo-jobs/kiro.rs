## 缓存方案对比

### 方案 1：简单缓存（Exact Match）

**原理：** 完全匹配请求参数（prompt + model + temperature 等）

**优点：**
- ✅ 实现简单
- ✅ 性能高（O(1) 查找）
- ✅ 无需额外依赖

**缺点：**
- ❌ 命中率低（参数必须完全一致）
- ❌ 无法处理语义相似的请求

**适用场景：**
- 重复的固定 prompt
- 测试环境
- 文档生成等固定模板场景

**实现难度：** ⭐⭐☆☆☆（简单）

---

### 方案 2：语义缓存（Semantic Cache）⭐ 推荐

**原理：** 使用向量相似度匹配语义相似的请求

**优点：**
- ✅ 命中率高（语义相似即可命中）
- ✅ 节省成本显著
- ✅ 降低延迟

**缺点：**
- ⚠️ 需要 Embedding 模型
- ⚠️ 需要向量数据库（Redis/Qdrant）
- ⚠️ 实现复杂度中等

**适用场景：**
- 生产环境
- 用户对话场景
- FAQ 系统
- 客服机器人

**实现难度：** ⭐⭐⭐☆☆（中等）

---

### 方案 3：Prompt Cache（上游缓存）

**原理：** 利用 Anthropic 的 Prompt Caching 功能

**优点：**
- ✅ 官方支持
- ✅ 无需本地实现
- ✅ 节省 Token 成本

**缺点：**
- ❌ 仅缓存 prompt 前缀
- ❌ 需要上游 API 支持
- ❌ 无法完全避免 API 调用

**适用场景：**
- 长系统 prompt
- RAG 场景（知识库前缀）
- Agent 工具定义

**实现难度：** ⭐☆☆☆☆（简单，仅需传递参数）

## 推荐实现方案

### 🎯 混合方案：简单缓存 + 语义缓存

#### 架构设计

```
┌─────────────┐
│   Client    │
└──────┬──────┘
       │
       ▼
┌─────────────────────────────────────┐
│         Axum Middleware             │
│  ┌───────────────────────────────┐  │
│  │   1. 简单缓存检查（Redis）     │  │
│  │      - 完全匹配 → 返回缓存     │  │
│  └───────────────────────────────┘  │
│               │ Miss                │
│               ▼                     │
│  ┌───────────────────────────────┐  │
│  │   2. 语义缓存检查（Qdrant）    │  │
│  │      - 相似度 > 0.85 → 返回   │  │
│  └───────────────────────────────┘  │
│               │ Miss                │
│               ▼                     │
│  ┌───────────────────────────────┐  │
│  │   3. 调用上游 API             │  │
│  │      - 获取响应               │  │
│  │      - 存入两级缓存            │  │
│  └───────────────────────────────┘  │
└─────────────────────────────────────┘
```

#### 技术选型

1. **简单缓存：** Redis
   - Key: `SHA256(request_json)`
   - Value: `response_json`
   - TTL: 1 小时

2. **语义缓存：** Qdrant（或 Redis Vector）
   - Embedding: `text-embedding-3-small`
   - 相似度阈值: 0.85
   - TTL: 24 小时

3. **Rust 依赖：**
   ```toml
   redis = { version = "0.24", features = ["tokio-comp", "connection-manager"] }
   qdrant-client = "1.7"
   ```

## 风险和注意事项

### ⚠️ 需要注意的问题

1. **流式响应缓存**
   - 需要完整接收响应后才能缓存
   - 首次请求延迟不变
   - 后续请求可以模拟流式返回

2. **缓存失效策略**
   - 模型更新后需要清理缓存
   - 用户反馈不准确时需要清理
   - 定期清理过期缓存

3. **隐私和安全**
   - 敏感信息不应缓存
   - 需要用户级别的缓存隔离
   - 考虑加密存储

4. **一致性问题**
   - 相同请求可能返回不同结果（temperature > 0）
   - 需要配置哪些参数影响缓存 key
